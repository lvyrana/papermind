# 产品决策记录

PaperMind 的每一个决策都来自"为什么这样做"的思考过程，而不仅仅是"做了什么"。本文档记录了所有重要的产品和技术决策，按时间线组织。

---

# 核心设计理念

PaperMind 的每一个决策都来自这些核心理念。以下记录了产品价值观和设计思路。

---

## 为什么叫 PaperMind

项目最初叫 copd-research-weekly，是给 COPD 领域做的周报工具。后来意识到关键词和领域都是配置化的，任何学科都能用。PaperMind 取 Paper（论文）+ Mind（记忆/思维），一词包含了文献追踪和对话记忆两个核心能力。

## 为什么首页叫"研究空间"而不是"论文列表"

普通学术工具一上来就是搜索框+表格，很工具感。但研究者打开一个每日助手，应该有被理解的感觉。所以首页用问候语（"早上好，研究者"）+ 上次在读 + 为你推荐，营造"你的研究伙伴已经帮你准备好了"的感觉。

## 为什么要做"研究者画像"

没有画像，推荐就是纯关键词匹配。有了研究方向、排除领域、当前目标，AI 能写出"这篇论文的 XX 方法可以迁移到你正在做的 YY"这种真正有用的解读。这不是功能堆砌，是让推荐从"相关"变成"有启发"。

## 为什么推荐结果要写"为什么与你相关"

学术工具给你 50 篇论文让你自己判断，但研究者时间有限。一句"这篇和你的研究方向的交集在于……"能帮你在 3 秒内决定要不要读。这是从"给你信息"到"帮你判断"的跨越。

## 为什么做对话式阅读而不是只给摘要

摘要是单向的。但读论文时真正的思考发生在追问中："这个方法靠谱吗？""样本量够吗？""和我做的有什么区别？"。对话式阅读把 AI 变成一个懂论文内容、也懂你研究背景的讨论伙伴。

## 为什么对话要有记忆

因为研究是连续的。你上周和 AI 讨论了一篇方法学论文，这周读到相关论文时，你不想从零开始解释自己的背景。对话记忆让每一次讨论都能积累，让 AI 越来越了解你。

## 为什么整体风格选温暖而不是科技炫酷

市面上的学术工具几乎全是冷色调+极简工具风。但科研是孤独的，尤其是独立研究者。PaperMind 用暖色系（cream/coral/mint）、圆角卡片、手感动画，想传达的是：这不只是一个工具，是一个陪你读论文的温柔空间。

## 为什么加呼吸感动画

论文列表如果一次性全部出现，信息量会让人焦虑。逐条呼吸入场（淡入+上移+微模糊）让加载过程像翻书一样，一篇一篇出来，节奏从容。这是从"数据展示"到"阅读体验"的细节。

## 为什么导出到 Zotero 而不是内置文献管理

PaperMind 的定位是"发现+理解+思考"，不是要替代 Zotero。研究者已有的文献管理流程不应该被打断。一键导出 RIS/BibTeX 让 PaperMind 成为现有工作流的入口，而不是孤岛。

## 为什么新用户引导要以"记忆"而不是"搜索"为核心卖点

测试反馈显示新用户打开首页完全不知道这是什么。我们有两种方向可以讲：一是"帮你搜论文"（功能层），二是"有记忆的助手"（价值层）。搜论文工具太多，但一个"告诉它一次、永久记住你、越用越懂你"的助手是有差异化的。引导页改为以"有记忆的学术文献助手"为标题，三个价值点也围绕记忆展开：记住你是谁 / 每天帮你过滤 / 沉淀你的思考。

## 为什么在画像未填时锁住"重新抓取"而不是只提示

第一批测试中有用户在画像空着的情况下连续点"换一批"，把一天的推荐配额（8批）全部消耗，却一篇论文没看到。根本原因：没有画像时检索词质量极差，大概率 0 结果或返回无关论文，但系统仍然扣配额。决定直接在 UI 层拦截：画像未填时隐藏"重新抓取"，空状态按钮变为"先填写研究方向"跳转 Profile。硬拦截比软提示更能防止这种情况。

## 为什么用专属链接而不是账号体系来支持跨设备访问

测试用户在电脑端测完后想在手机继续用，但现有设备 ID 机制要求手动复制粘贴一长串 UUID。登录体系需要邮箱/密码，成本高且目前用户量小。折中方案：生成带 uid 参数的专属链接（`?uid=xxx`），收藏书签或发给自己即可在任何设备无缝继续，无需理解技术细节。

## 为什么在后端而不只在前端拦截空画像的推荐请求

测试截图显示用户在画像为空的状态下触发了 8 次推荐抓取，每次 0 组查询、0 篇结果，但 8 次配额全部被扣光。前端引导页（画像空时显示 Onboarding）是第一道防线，但它只在"无缓存论文"时生效，存在绕过路径（如缓存被清、localStorage 问题）。后端守护才是最后的保障：在即将 `increment_rate_limit` 之前检查画像是否有任何实质内容，若完全为空则拒绝并返回 `needs_profile: true`，配额不动。两道防线互补：前端负责引导体验，后端负责配额安全。

## 为什么 LibraryDetail 用轮询而不是 WebSocket 等待 AI 解读

手动添加论文后，后台 AI 解读需要 5-15 秒完成。WebSocket 实现复杂、部署需改 nginx 配置。简单轮询（4s 间隔，有结果即停）足够解决这个低频场景，且实现成本极低。加"AI 解读生成中…"占位是关键——没有它，用户看到空白区域会以为功能出错。

## 为什么用模式匹配清除 localStorage 而不是维护一个固定列表

`paper-notes-*`、`paper-chat-*`、`paper-bookmark-*` 的 key 是动态拼接 pmid/paper_id 的，数量不定。维护一个固定列表意味着每次新增 localStorage key 都要同步更新清理列表，容易遗漏。改为遍历 localStorage 全量匹配前缀，一次清干净，且后续加新 key 不需要改清理逻辑。

## 为什么解读轮询超时选 60s 而不是更长

AI 解读单篇论文正常耗时 3-10s，即使后台排队也不应超过 30s。60s（15 次 × 4s）已留有充足余量。超过这个时间说明后台大概率已失败，继续轮询只会浪费请求、让用户看到永不停止的转圈。超时后静默停止比报错更合适——用户可以稍后刷新重试。

## 为什么对话输入框加麦克风而不是独立的语音入口

用户在手机上和 AI 讨论论文时打字慢，加语音是刚需。把麦克风按钮直接放在输入框旁边（而不是单独页面或悬浮按钮），视觉上和文字输入并列，符合"任意时候都可以说"的使用预期。识别结果追加到已有输入框内容（而不是覆盖），是为了让用户可以先打几个字再用语音补全，混合使用不丢文字。不支持的浏览器隐藏按钮而不是禁用，防止用户误以为是 bug。

## 为什么删掉设置页的"设备 ID"卡片

设备 ID 卡片原来的价值是让用户复制 UUID 去另一台设备粘贴以实现数据迁移。"多端同步"专属链接功能上线后，这个流程已被完全替代——用户只需点一下复制链接，不需要理解 UUID 是什么。保留裸 UUID 反而会让不熟悉技术的用户感到困惑（"这串字符要用来干嘛？"）。设备绑定的说明文字保留在多端同步卡片的描述里，信息不丢。

# 产品决策记录

记录 PaperMind 开发过程中的重要产品决策，包括选择的方案、放弃的方案和原因。

---

## 2026-04-17（续）

### UID 持久化选用 localStorage + cookie 双写，不内嵌固定值
**决策**: 不把 `OWNER_UID` 打包进前端构建，而是在 `getUserId()` 里同时写入 localStorage 和 cookie，并在设置页提供手动输入同步码的入口。
**原因**: 若将 UID 硬编码进构建产物，所有测试用户都会发送同一个 UID，导致数据混用、限流规则失效。单人 app 的"owner 特权"只应存在于服务端环境变量，不应影响前端身份标识。
**取舍**: cookie 在 Safari / 移动端的持久性远优于 localStorage；双写方案兼顾兼容性与稳定性，同步码入口给 owner 提供跨设备切换能力，普通用户不受影响。

## 2026-04-11

### 时间范围必须以最终论文结果为准，而不是只靠数据源参数近似
**决策**: 用户在画像里选择的时间窗，最终必须在后端结果集上再按真实日期过滤一次，不能只依赖数据源侧的粗粒度参数。  
**原因**: Semantic Scholar 只有年份级过滤时，很容易让“近一个月”混入去年的论文。对用户来说，这会直接破坏对推荐系统的信任。  
**取舍**: 即使数据源已有时间参数，后端也要再做一层真实日期过滤，优先保证用户感知的一致性。

### PubMed 长查询优先保留短语语义，不追求过度精确
**决策**: 对较长的英文检索词组，不再一律拆成超长 `AND [tiab]` 链，而是优先保留短语匹配，并只在必要时用少量 anchor terms 兜底。  
**原因**: 真实日志证明，过长的 `AND` 链会让 PubMed 在短时间窗下大量返回 0 结果，最终导致推荐结果几乎全由其他来源接管。  
**取舍**: 先让召回池更稳定，再靠后续排序和过滤控制质量，而不是在第一步就把查询收得过死。

### 页面标题层级优先统一，少用重复标签
**决策**: 收藏页和研究画像页的标题栏统一字号、字重和上下边距；重复表达同一层含义的小标签（如英文装饰标签、重复的分区小字）优先去掉。  
**原因**: 当前产品更需要稳定、自然的阅读节奏，而不是靠多层标签制造“设计感”。当标题、分区标题、字段标题都在重复说差不多的事时，页面会显得拥挤而不清楚。  
**取舍**: 保留真正有信息量的层级，把装饰性标签降到最低，让页面更像一个完整产品而不是多个草稿页拼起来。

### 收藏详情里的长笔记编辑优先照顾手机场景
**决策**: `✨ 总结` 这类长笔记进入编辑态时，默认使用更高的 textarea，并允许继续纵向拉伸；编辑 / 删除按钮在移动端默认可见。  
**原因**: 测试过程中，手机上进入编辑后只露出前几行内容，既不方便修改，也找不到删除入口。对 PaperMind 这种以手机回看和随手调整为主的测试版来说，移动端可操作性优先于桌面端的极简 hover 效果。  
**取舍**: 桌面端仍保留较轻的 hover 反馈，但移动端不再隐藏关键操作。

### 同一阿里云 Key 下优先做多模型顺序回退
**决策**: 不为每个千问模型单独写死一套分支，而是在同一个 `QWEN_API_KEY` 下，通过 `QWEN_MODEL + QWEN_FALLBACK_MODELS` 配置一条顺序回退链。  
**原因**: 当前主要目标不是“升级到更强模型”，而是把阿里云赠送的多组免费额度尽量串起来，延长 PaperMind 的可用时间；用配置而不是硬编码，后续换模型顺序时只改 `.env` 就够了。  
**补充**: 后端日志会打印每次实际命中的 provider / model，方便在线确认当前到底用了哪一个模型。

### 首页问候语先保持匿名，等待登录体系再个性化
**决策**: 首页问候语现阶段只显示“早上好 / 下午好 / 晚上好”，不再使用“研究者”作为默认称呼。  
**原因**: 测试版还没有登录体系，也没有稳定的昵称来源；强行加一个固定称呼会显得有点生硬。等以后真正加入登录或个人昵称后，再自然升级成“早上好，某某”。  
**取舍**: 先保留轻量、自然的欢迎语，把个性化留给后续真正有身份信息时再做。

### 收藏动作需要迁移未收藏阶段的本地对话
**决策**: 当用户在首页详情页先与论文对话、后续再点击收藏时，收藏动作必须把本地临时对话一并迁移到后端。  
**原因**: 用户会自然地先聊一会儿再决定是否收藏，如果收藏只保存论文而不迁移旧对话，进入收藏详情后会误以为聊天记录丢失。  
**实现**: 收藏接口允许接收当前临时对话；仅在该论文还没有后端聊天记录时执行一次迁移，避免重复写入。

### 对话总结只保留服务端那条结构化笔记
**决策**: 首页详情页执行“将对话保存为笔记”时，不再把总结内容再拼接进自由笔记文本，只保留后端保存的 `chat_summary`。  
**原因**: 服务端已经会落一条独立总结笔记，前端再额外追加到自由笔记区，会制造一条重复的“手写笔记”，用户感知上就是生成了两份相同总结。  
**取舍**: 让“自由笔记”和“AI 总结笔记”保持两套明确语义，避免一条内容以两种来源重复出现。

### SQLite 先保留单机形态，但补上日备份
**决策**: 在正式迁移数据库前，继续使用单机 SQLite；同时在 ECS 部署层补一套最小日备份方案，而不是立刻引入更重的数据库服务。  
**原因**: 现阶段是小规模测试版，SQLite 足够支撑；真正高风险的是“只有一份数据”。先把备份补齐，比现在就迁数据库更能降低风险。  
**实现**: 用 SQLite 原生 `.backup` 生成每日备份，配合 `systemd timer` 每天自动执行，并保留最近 14 天。

### 设置页设备 ID 读取失败时，降级显示而不是整页白屏
**决策**: 设置页不再在首次渲染时“强依赖”本地设备 ID 成功读取；如果移动端浏览器无法访问 `localStorage` 或不支持标准 UUID 能力，页面仍应正常显示，并提示当前环境暂时无法读取设备 ID。  
**原因**: 测试版会在手机 Safari / Edge 等真实浏览器中使用，设备 ID 是辅助信息，不应该因为这一项失败就让整页不可用。  
**实现取舍**: `getUserId()` 增加内存兜底；设置页挂载后再读取，失败时只降级设备相关操作，不影响页面浏览。

### AI 连通性测试仅对 owner 开放
**决策**: `/api/settings/test` 不对普通测试用户开放，只允许 `OWNER_UID` 对应设备调用；若服务端未配置 `OWNER_UID`，返回明确部署提示。  
**原因**: 这是调试性质接口，放开会平白消耗 token，且容易被误当成普通功能入口。  
**补充**: 普通用户的 AI 使用仍通过首页推荐、翻译和阅读页对话完成，不需要暴露额外测试接口。

### 测试期默认限额收紧为 8 / 20 / 30
**决策**: 默认限额调整为“每天最多获取 8 批推荐结果、20 次 AI 对话、30 次翻译”。  
**原因**: 对测试用户来说，这个额度足够覆盖真实试用；同时“批次”比“次数”更准确，能减少对推荐额度的误解。  
**实现**: 后端默认值、`.env.example` 和设置页说明保持一致，避免线上与文案脱节。

### AI 限速改为“端点限额 + 全局熔断”
**决策**: 不再只保留单一的论文推荐次数限制，而是按端点拆分：推荐、对话、翻译分别限额；对 AI 对话与总结额外增加全局每日熔断。  
**原因**: 真实使用里，对话和翻译的成本结构与推荐不同。分开控制更符合测试期的资源管理，也更容易在 `.env` 中按实际消耗调参。  
**补充**: owner 设备豁免用户级对话/翻译限额，但不豁免全局熔断，避免单设备把当日总额度耗尽。

### 笔记导出优先做 Markdown，而不是更复杂的同步集成
**决策**: 第一版“全部导出”选择 Markdown 下载，不直接做 Obsidian/Notion API 同步。  
**原因**: Markdown 是最通用、最低锁定的中间格式，既能本地保存，也方便后续导入别的知识管理工具。  
**取舍**: 先解决“把自己的想法带出去”的核心需求，第三方同步能力以后再接。

### ECS 部署优先提供脚本化最小方案
**决策**: 针对 Ubuntu 22.04 + 阿里云 ECS，仓库内直接提供 `setup.sh`、`update.sh`、systemd 与 nginx 配置。  
**原因**: 当前项目由个人维护、迭代频繁，比起一开始引入更复杂的容器编排或多平台部署，先把“能稳定部署并更新”做成一键脚本更符合现阶段需求。  
**代价**: 这套方案偏单机，后续如果迁移到更正式的云架构，还需要再抽象部署流程。

### 系统观察摘要退出搜索召回，只保留在手动修正后的理解层
**决策**: `interests_summary` 不再参与关键词生成；论文打分和 AI 解读只在 `interests_summary_is_manual = '1'` 时参考“用户修正后的偏好”。  
**原因**: 系统观察摘要来自行为归纳，容易很散，适合帮助模型理解用户，但不适合主导“搜什么”。如果把它接进检索召回，系统容易把“看过很多”误当成“现在要搜很多”，导致方向发散。  
**补充**: 用户手动编辑或手动清空摘要后，保存画像不会立刻触发自动重生成，避免用户修正被系统覆盖。

### 收藏页改为紧凑行列表，优先支持“回看我的笔记”
**决策**: 收藏页不再按时间大卡片分组展示，改为更紧凑的单列行列表，并增加“有笔记”筛选；删除按钮在移动端保持可见。  
**原因**: 当前阶段用户的核心任务不是“浏览漂亮的论文墙”，而是“快速找到自己写过笔记、想回看的内容”。大卡片和 hover-only 操作在手机上都不够顺手。  
**取舍**: 先优先回看效率和移动端可操作性，知识图谱/灵感库形态留待后续单独视图探索，而不是继续叠加在收藏列表上。

### 研究画像改为“长期偏好 + 系统观察”工作台
**决策**: 不再维持旧版逐项表单，改成新的研究画像工作台：研究方向、方法兴趣、随手补充、排除内容、学科领域，加上一块可查看和编辑的“系统观察摘要”。  
**原因**: 用户反馈旧画像页“像设置表单，很难写”，尤其研究经历和追踪主题不容易维护。新结构把长期偏好和系统自动学习结果放到同一页，填写成本更低，也更接近真实研究工作流。  
**代价**: 页面结构更复杂，字段语义需要后端同步梳理；如果不及时清理旧字段，容易出现前端不再暴露但后端仍在读取的历史状态。

### 收藏页回归 v0.5 信息结构
**决策**: 收藏页本轮不继续往“复杂灵感库”方向扩张，回到更接近 v0.5 的信息结构。  
**原因**: 用户明确反馈“内容更多了也更丑了，看笔记也不方便”。在当前阶段，比起继续加功能，更重要的是保住可读性和回看笔记的效率。  
**取舍**: 先稳定收藏体验，知识图谱/灵感库能力留待后续以独立视图或模式切换的方式尝试，而不是直接叠加在收藏列表上。

### 用户归属校验改为严格匹配，不再兼容空 user_id
**决策**: 收藏详情、删除收藏、保存笔记、AI 对话、对话总结等接口统一改为严格按 `user_id` 校验归属。  
**原因**: 兼容空 `user_id` 的旧策略虽然能照顾早期测试数据，但会让多人测试环境下的笔记、对话、导出都存在串数据风险。既然项目已经建立了设备级 UUID 隔离，后续实现应围绕严格归属校验收紧。  
**补充**: 为避免旧数据丢失，先做本地历史数据认领迁移，再切到严格校验。

### 搜索策略先回到 v0.5 主路径，再增量接入方法兴趣
**决策**: 放弃一度尝试的“方法学扩展宽搜”方案，回到 v0.5 的主题检索主路径，只把 `method_interests` 作为搜索词生成和排序的辅助输入逐步接入。  
**原因**: 纯方法学召回和过宽兜底会让结果快速发散，用户实际反馈是“只想在原有体感基础上稍微参考方法兴趣”，而不是完全重写召回逻辑。  
**代价**: 当前 `method_interests` 已参与搜索，但权重仍需继续调；时间范围在 Semantic Scholar 路径上仍然是宽松近似，不是严格月窗。

### 标题翻译与摘要翻译保持一致
**决策**: PaperRead 页的论文标题加"中文/原文"切换按钮，样式与摘要翻译按钮一致（文字小、低调）。  
**原因**: 用户反馈"进入论文页后标题没有翻译选项"；保持交互一致性，不引入新的 UI 模式。  
**细节**: 首次点击时懒加载翻译，之后切换仅切换显示状态，不重复请求。

### Legacy 论文 ownership 检查兼容策略
**决策**: ownership 检查改为 `if paper.get("user_id") and paper.get("user_id") != uid` 而非 `paper.get("user_id", "") != uid`。  
**原因**: 早期收藏的论文 `user_id` 字段为空字符串，旧检查将其视为"属于别人"，导致 GET/DELETE/notes 全部返回 not found。兼容写法：有 user_id 才做归属校验，没有则视为公共数据可访问。  
**代价**: 没有 user_id 的论文任何用户都能访问。可接受，因为这些都是早期测试数据。

### 对话失败反馈：显式错误而非静默失败
**决策**: "将对话保存为笔记"按钮的所有失败路径（自动收藏失败、API 错误、网络错误）均显示具体中文错误信息，不再 catch 后忽略。  
**原因**: 按钮点击后没有任何反应，用户不知道是成功还是失败；显式错误让用户知道该怎么做（如"请先手动收藏"）。

### Library 搜索与筛选：前端实现，不走后端
**决策**: Library 页的全文搜索、分类筛选、排序全部在前端完成，不新增后端接口。  
**原因**: 收藏数量有限（几十到几百篇），前端过滤够用；不值得为此加后端复杂度。  
**代价**: 超大收藏量时性能会下降，但对个人研究工具场景不是问题。

### Profile 两层结构：基础信息 vs 当前专注
**决策**: Profile 页拆为"基础信息"（相对稳定：研究方向、阶段、经验、学科）和"当前专注"（频繁变化：研究问题、目标、排除领域）两块。  
**原因**: 用户反馈画像维护成本高，一屏全部展示容易混淆稳定信息和动态信息。分层后用户知道"当前专注"要经常更新，"基础信息"不常动。

## 2026-04-27

### 用户等待路径全面 async 化，不再阻塞 FastAPI 事件循环
**决策**: 所有有用户等待 AI 返回的接口改为 `async def`，LLM 调用改用 `AsyncOpenAI`；后台线程继续用同步桥接（`asyncio.run(_llm_chat_complete_async(...))`）。
**原因**: 之前的同步路径下，一个用户请求对话，另一个用户的翻译就会在线程池里排队，表面看起来像"后台卡住"，实际是 Python 线程占满事件循环。async 后多个请求能真正并发执行。同时，provider 路由和冷却逻辑只维护一份（在 `_llm_chat_complete_async` 里），同步桥接和异步路径不再各写一套。
**取舍**: 后台抓取 / 批量解读因为在 `ThreadPoolExecutor` 里跑，暂时保留同步桥接，第一阶段优先改掉"用户感知最强"的接口：对话、翻译、记忆、设置测试。

### 导航栏统一成固定顶栏，移除移动端浮动底栏
**决策**: 移动端浮动底栏（`fixed bottom-6`）和桌面端共用同一套固定顶部导航栏，底栏代码以 `{false && ...}` 方式停用。
**原因**: 移动用浮动底栏、桌面用顶栏，是两套不同的导航模型，维护成本和样式冲突都更高；单一顶栏在所有尺寸保持一致体验，实现也更简单。
**代价**: 移动端失去部分"拇指友好"的底部触达；顶栏在小屏上位置偏上，需要一定适应期。

### Profile 桌面端：保存按钮移到侧边栏，检索范围独立成卡片
**决策**: 把"保存画像"按钮从主区底部迁移到左侧边栏底部；把检索时间范围从侧边栏小组件里抽出，在主区独立成一张完整卡片。
**原因**: 侧边栏是"画像工作台"的只读快照区，保存按钮放在那里更符合"编辑区 vs 操作区"的职责划分。检索时间范围在侧边栏里按钮太密，独立成卡片后可以显示完整标签（"近1个月"而不是"近1月"），也更容易选中。
**取舍**: 保存按钮不在主区底部了，部分用户需要适应新位置；但 coral 色 + 固定在侧边栏底部，视觉上仍然突出。

### 页面过渡缓存：先从 localStorage 渲染，再等 API 数据
**决策**: Profile 页进入时先从 `localStorage` 读取上次缓存的画像数据立即渲染，API 数据回来后再无感更新；每次状态变更都同步写入 `localStorage`。
**原因**: 用户点击进入画像页，若等 API 才渲染，会看到一片空白然后"跳"出内容，体验很差。本地缓存让页面在 0ms 内就显示上次状态，API 返回时 quietly 更新。
**代价**: 如果后端数据在另一台设备上有更新，进入画像页会短暂显示旧数据。对单人工具来说几乎不发生。

## 2026-04-23

### Provider 冷却机制：401/403 耗尽后自动跳过 10 分钟
**决策**: 当某个 LLM provider 返回 401/403/quota 相关错误时，标记冷却 10 分钟，期间直接跳过该 provider，不再每次请求都先撞一次。
**原因**: 之前无冷却机制，一旦某个模型配额耗尽，后续每条请求都会先尝试（报 401）再尝试下一个，150 篇论文 = 150 次无意义的错误调用。加冷却后，一旦检测到配额耗尽，立刻跳过，只浪费一次请求。

## 2026-05-04

### 论文详情标题拆成中英文两套样式
**决策**: 论文详情页标题不再用一套字体同时承载中英文，而是按当前显示语言切换两套 class。
**定稿值**:
- 英文标题：`"Iowan Old Style", "Palatino Linotype", Palatino, Georgia, serif`
- 英文字重：`600`
- 英文字号：`23px`
- 英文行高：`1.5`
- 英文颜色：`#274A73`
- 中文标题：`"Songti SC", "STSong", "Noto Serif SC", "Source Han Serif SC", "SimSun", serif`
- 中文字重：`600`
- 中文字号：`23px`
- 中文行高：`1.6`
- 中文颜色：`#274A73`
**原因**: 之前中英文共用一套 serif，浏览器和系统 fallback 不稳定，容易出现中文偏细、英文偏硬、不同设备观感变化大的问题。拆开后，各自用更适合自己的字体栈，显示更稳定。
**补充**: 标题翻译按钮增加失败提示，避免后端未启动或模型异常时静默无反应。
**效果**: 配额耗尽时的整体耗时减少 30%+，后端日志更清爽，用户感知延迟显著降低。
**实现**: `_is_quota_error()` 识别 401/403/quota/rate_limit 关键词；`_provider_cooldown` 字典维护冷却截止时间；`_llm_chat_complete()` 和 `_get_llm_client()` 在遍历 provider 时检查冷却状态。

### 系统观察摘要触发策略：改为事件阈值 + 时间窗口
**决策**: 不再"每次进首页都尝试生成"或"画像一变就立刻重生成"，改为：
- 累计关键行为 ≥ 8 次（收藏、对话、阅读任一行为）触发重生成，OR
- 距上次生成 ≥ 7 天且有新行为触发
- 用户手动点"更新"按钮
**原因**: 之前的 24h 冷却容易被画像变化清空，导致频繁重生成；新方案让摘要稳定作为"一周一总结"的节奏，同时累计足够行为数据再更新，质量和成本都更稳定。
**数据库**: `user_profiles` 新增 `behavior_events_since_summary` 字段记录自上次摘要生成后的行为数。每次收藏、对话、阅读时 `increment_behavior_events(uid)`；摘要生成后 `reset_behavior_events(uid)` 归零；画像变化时同步重置。
**体验**: 用户在测试期间操作不会频繁触发摘要重生成，降低 token 消耗焦虑；摘要内容相对稳定，质量也更可控。

### 摘要生成指定 Qwen 特定模型，保证质量一致性
**决策**: 系统观察摘要的 LLM 调用新增 `prefer_model="qwen-flash-2025-07-28"` 参数，确保这个任务总是用指定模型，其他任务正常轮询。
**原因**: 用户反馈换 qwen3.5-flash 后摘要质量变差；摘要是对用户的"长期理解"，质量稳定性很关键。用 qwen-flash-2025-07-28（更新、更好的开源模型）确保摘要质量，同时不影响其他任务的冷却轮询。
**实现**: `_llm_chat_complete()` 新增 `prefer_model` 参数，将指定模型排在 slots 最前面；如果该模型配额耗尽，自动 fallback 到其他 provider，无需额外处理。

### Enrichment 缓存到 SQLite：同一篇论文不重复调 LLM
**决策**: 新增 `enrichment_cache` 表，缓存论文的 `summary_zh / relevance / key_findings`；同一 PMID/DOI 第二次出现时直接命中缓存，跳过 LLM 调用。
**原因**: P0 并行化抓取后，同一篇论文容易在多轮推荐中重复出现；之前每次都调 LLM 浪费 token；现在命中缓存直接 0s。
**缓存键**: `pmid:{id}` 优先，其次 `doi:{id}`，都不存在则不缓存。
**效果**: 重复论文的解读耗时从 3-10s 降到 0s；库累积越多、重复率越高、收益越明显。

### 分类打分并行化：8 批顺序 → 4 线程并行
**决策**: `categorize_papers.py` 的 `_score_batch()` 改为线程池并行调用，4 个 worker。
**原因**: 之前 146 篇论文分 8 批，每批调一次 LLM（3s），串行共 21s；并行化后 ~6s，整体推荐流程从 48s 降到 ~30s。
**实现**: 用 `ThreadPoolExecutor(max_workers=4)` 和 `as_completed()` 对批次列表并行，异常忽略。
**限制**: 4 个 worker 是经验值，对 token limit 影响小（每批独立请求），不会导致回复断裂。

### Rate limit 重新定义：只对「重新抓取」计费
**决策**: `force_fetch=true`（用户主动点"重新抓取"）才消耗推荐配额；缓存重建（服务重启、缓存过期、首次加载）不消耗。
**原因**: 测试发现服务重启 8 次，前端轮询自动触发 8 次新抓取，用户实际只点了 1 次搜索，但配额吃完了；痛点是"自动触发 = 无意义消耗"。
**效果**: 不再对幽灵消耗焦虑；配额只反映真实用户意图。

## 2026-04-24

### 后端并发：workers 2 + SQLite WAL，而不是全面 async 改造
**决策**: 先将 uvicorn workers 从 1 改为 2，并开启 SQLite WAL 模式，暂不做全量 async LLM 改造。
**原因**: LLM 调用是同步阻塞的（httpx.Client），单进程线程池约 5 个线程，多个 LLM 调用同时进来会耗尽线程池，导致翻译等用户可见操作排队。workers 2 是最小改动，能立即缓解；WAL 模式让两个进程并发写 SQLite 不互相锁死。
**为什么不先做 async**: async 改造涉及整条调用链（`_llm_chat_complete` → 所有 helper → 所有路由函数），改动面大，作为后续独立任务规划。
**后续**: 计划在独立分支做 AsyncOpenAI + async 路由函数改造，作为彻底解法。

### 中文字体：system-ui 优先，Noto Sans SC 作兜底
**决策**: font-sans 字体栈顺序为 `DM Sans → system-ui → Noto Sans SC → sans-serif`。
**原因**: system-ui 在 Mac/iPhone 上解析为苹方（视觉效果好、用户习惯），在主流 Android 上也有完整 CJK 覆盖。Noto Sans SC 放在 system-ui 之后，只在系统字体 CJK 覆盖不全时兜底（某些 Android 设备）。若 Noto Sans SC 放前面，Mac 桌面端会从苹方切换为 Noto Sans SC，字形偏紧凑，用户反馈明显。

### 记忆 recent 的长度约束：prompt + 硬截断双保险
**决策**: recent 生成用 prompt 指定 100-180 字，同时在代码层加 `_enforce_recent_length(180)` 硬截断。
**原因**: 只靠 prompt 约束不可靠，模型有时会忽略字数限制；recent 的定位是"轻量提醒"而非完整摘要，过长会和 core 高度重复，失去两层分离的意义。硬截断是最后一道防线，确保 recent 始终短、轻。

### quote 成为一等阅读对象，先用矩形 anchor 解决“刷新后还在”
**决策**: 新增 `paper_quotes` 表，把划词文本、页码、页面相对矩形、追问、回答和来源结构化保存；前端 PDF 每页增加 highlight layer，根据 quote anchor 重画高亮。
**原因**: 旧方案只把 quote 暂存在前端 chat message 的 `_quote` 字段里，刷新、跨设备、后端聊天恢复后都容易丢；用户看到“划选了但右栏没接住”，本质是 quote 没有成为后端对象。
**取舍**: 第一版 anchor 采用 PDF 当前渲染页的相对矩形，并记录生成时 scale，缩放时按比例重画。这比“全文搜索原文再定位”稳定、工程量小，也能马上解决视觉痕迹持久化；代价是如果 PDF 文本层排版算法或页面裁切变化很大，旧 anchor 可能轻微偏移。后续再叠加 textStart/textEnd 或全文字符 offset 做二次校准。

### 精读动作也应留下引用痕迹，而不只聊天追问
**决策**: 「问 paperMind」会在聊天成功后保存带 question/answer 的 quote；「精读这段」和「存为卡片」也会保存 quote anchor，只是来源分别记为 `deep_read` / `card_seed`。右栏文案从“你在这篇里追问过”改为“引用与追问”。
**原因**: 精读不是只有聊天。用户真正读懂一篇文章时，会做三类动作：追问、拆解、沉淀。只保存追问会漏掉“精读这段”和“存卡”这两类高价值阅读痕迹。
**边界**: 当前 quote 删除只提供后端 API，前端还没有做单条删除按钮；后续如果引用变多，需要在右栏加删除/筛选，并把 quote 与卡片、PPT 汇报板块建立更明确的关系。

### 精读右栏只承载阅读动作与沉淀，不再混放元信息和画像预告
**决策**: PaperRead 右栏移除空的「你在这篇里追问过」占位、「这篇会让画像往这里走」预览和「标题翻译 · 摘要」折叠区；摘要回到左侧元信息栏，quote 归档只在用户真正发出带引用的追问后出现。  
**原因**: 当前主线是“带着英文吃力的用户精读一篇文章”。右栏应该服务于当下阅读、追问、带读、卡片和笔记沉淀；标题/摘要属于元信息，画像预告没有真实闭环，空 quote 占位又会误导用户以为“只要划选就会自动进入这里”。  
**边界**: 这不是否定标题翻译、画像学习或 quote 历史，而是先把它们从精读主通道拿出去。后续如果要做专业阅读器式划词体验，应实现稳定的 quote anchor、自定义高亮 overlay 和后端结构化 quote 存储，而不是继续依赖浏览器蓝色选区。

### 图表追问先用当前页文字上下文兜底，不假装拥有视觉理解
**决策**: `/api/chat` 接收当前页页码和 text layer 文本；当用户问 Fig/Figure/图/表时，模型优先根据当前页图注、标题和周围段落解释，同时明确这是基于文字上下文而非真实图像视觉识别。  
**原因**: 现阶段 PaperMind 还没有把 PDF 页面截图或图像区域送给视觉模型。直接回答“看不见图”体验很差，但假装看见图也会误导；当前页文字上下文是成本最低且诚实的中间态，尤其多数论文图注已经包含足够解释。  
**下一步**: 若要真正读图，需要做页面截图/图块裁剪、图注绑定和视觉模型调用，并在回答中区分“图像读数”和“图注推断”。

### 划词浮窗用视口坐标 + fixed，滚动时收起而不是跟随
**决策**: 浮窗定位从"滚动内容坐标 + absolute"改为"视口坐标 + fixed"；PDF 滚动时直接收起浮窗，不做跟随重定位。

### Zotero 插件 manifest：update_url 用 https 占位而不是删掉
**决策**: `applications.zotero.update_url` 指向 `https://127.0.0.1:8000/api/zotero-plugin/update.json`——一个本地开发阶段必然连不通的 https 地址。
**原因**: Zotero 9 有两条叠加的硬规则：① Zotero 自己的补丁要求 manifest 必须提供 `id` / `update_url` / `strict_max_version` 三项，缺一判无效；② Gecko 内核要求 updateURL 必须 `https:` 开头，否则插件被强制禁用。所以 update_url 既不能删也不能是 http，只能给一个 https 占位。更新检查连不通只是静默失败，不影响插件运行；未来后端上 https 后该地址自动变成真的。
**教训**: Zotero 的"可能无法与该版本的 Zotero 兼容"弹窗是所有安装失败共用的文案，与版本兼容毫无关系时也这么弹。排查这类问题不要信弹窗文案，用一次性 profile（`-profile` + 独立 dataDir）侧载插件、读 `extensions.json` 的 `appDisabled` 字段和 `-ZoteroDebugText` 日志，比猜快得多。

## 2026-07-04

### 自定义模型不做硬编码模型清单，做「从你的账号拉取」
**决策**: 设置页的模型选择不内置任何模型名列表，而是调用用户所填 API 的 `/models` 接口实时拉取。
**原因**: 模型名迭代极快，硬编码清单三个月就过时；而 `/models` 返回的是该 key 实际可用的模型，OpenRouter 这类聚合服务有几百个模型，实时拉取 + 关键词筛选比维护清单可靠得多。用户"不知道该用什么模型"的问题，答案应该来自她自己的账号，不是来自代码里的猜测。
**边界**: 模型输入框仍可手填，`/models` 不可用的服务不阻塞配置。

### 自定义通道排链首且不参与任务级模型重排
**决策**: 用户配置的自定义模型永远是 fallback 链第一位；现有的按任务（翻译/检索/对话）偏好模型重排只作用于内置链。
**原因**: 任务级重排是为内置通义多模型设计的优化；用户显式花钱配置的模型代表明确意愿，被系统"优化"到后面会造成"我配了却没在用"的困惑。失败回退内置链的行为保持不变，可用性不降级。
**技术注意**: 内置链的 httpx client 固定了 transport（绕过系统代理，国内服务直连），自定义通道必须用尊重环境代理的默认 client——OpenRouter 等国外服务在用户的网络环境下只能走代理。

### Zotero 集成选「单篇一键打开」而不是「Web API 批量导入」
**决策**: 做 Zotero 插件（右键「用 PaperMind 精读」），放弃把 Onboarding 里的 Web API 批量导入做成真的。
**原因**: 用户的真实动作是"在 Zotero 里决定精读这一篇 → 让它带着 PDF 出现在 PaperMind"——单篇、带附件、带意图。Web API 批量导入只有元数据没有本地 PDF，导入几百条不打算读的条目是库存噪声；它唯一的价值（冷启动画像）可以延后。
**实现要点**: 插件复用现有接口（/library/save + /library/{id}/pdf），后端零改动；浏览器侧用 `?uid=` 深链认领设备身份。

### `?uid=` 从"切换账号专用"改为"身份认领 + 深链"双语义
**决策**: `?uid=` 在 api.js 模块加载时同步认领；同 uid 保留路径不清缓存，不同 uid 才清缓存（原切换账号语义）。
**原因**: 旧实现的 UidHandler 无条件 `navigate('/')` 并清空缓存，任何带 uid 的深链都会被劫持回首页——Zotero 跳转链接根本无法到达阅读页。而"是否换了人"必须在模块加载时判断，因为等 effect 跑起来旧 uid 已被覆盖。
**教训**: 时序问题（React 子组件 effect 先于父组件）导致第一版修复失败，靠 Playwright 冷打开测试抓出来的——深链类功能必须用全新浏览器上下文验证，带缓存的开发环境测不出身份竞态。

### v1.0 大方向：从「发现」翻转到「精读与沉淀」
**决策**: v1.0 的产品重心从"每天帮你发现论文"翻转为"帮你读懂手上这一篇，并把读懂的东西变成能进工作流的产物"。发现流（每日推荐、配额、换一批）冻结不再迭代，精读闭环（引导式深读 → 卡片沉淀 → 报告/PPT/Obsidian 出口）成为主线。
**原因**: 作者进入文献精读阶段，真实需求是"一边精读一边产出阅读卡片/读书报告，方法学能沉淀复用"。v0.x 的推荐流是为多用户测试期设计的，与单用户精读场景错位。
**关键取舍**: 在现有代码基础上改而不是重写新产品——PdfViewer、划词交互、记忆系统、收藏/项目夹这些最厚的资产全部可复用，要新建的（卡片、导出）都是薄层。多用户时代的配额/限流/部署脚本冻结不删。

### 阅读卡片：四种类型对应精读的四个动作
**决策**: 卡片类型固定为方法/发现/批判/迁移四种，不做自由标签。
**原因**: 作者自述"精读小白，不知道怎么读、怎么把需要的东西留下来"。四种卡片类型本身就是精读方法论的产品化：方法卡对应"这研究怎么做的"，发现卡对应"结果是什么"，批判卡对应"哪里站不住"，迁移卡对应"我能怎么用"。类型即引导，自由标签反而回到"不知道记什么"的起点。
**边界**: AI 起草只出草稿，用户编辑定稿——内置 LLM 的方法学理解力有上限，最终判断必须留给人。

### 卡片必须挂在已收藏论文上，未收藏时自动收藏
**决策**: 复用对话总结的 ensureSaved 模式：保存卡片前若论文未收藏，先自动收藏再挂卡。
**原因**: 卡片是比笔记更强的沉淀信号，不应该因为"忘了点收藏"而丢失。且卡片游离于收藏体系外会让后续的导出/检索无从谈起。

### 本地 PDF 上传入口补全
**决策**: 阅读页无免费全文的空状态直接放"上传 PDF 精读"按钮；重进页面时本地 PDF 优先于开放获取查找。
**原因**: 后端上传接口 v0.8 就有但前端从未接线——而"手头有现成 PDF 要精读"恰恰是 v1.0 的第一场景。本地 PDF 优先是因为用户主动上传的版本（如出版社终版）通常好于 OA 渠道找到的预印本。

## 2026-05-21

### v0.8.0 大方向：从「AI 读论文工具」变成「有记忆的研究伙伴」
**决策**: 以"研究地形"为核心视觉隐喻，把记忆系统从配置角落抬升到每个页面的主舞台。
**原因**: 之前 memory_core / memory_recent 这些记忆数据被埋在 Profile 的一个小卡片里，用户感知不到 AI 在记着自己。这次重构的核心是"记忆要可见、可追溯、会生长"——用等高线地形图把研究方向可视化，让用户一进首页就能感受到 papermind 认识自己。
**关键取舍**: 表单（填写研究偏好）从 Profile 搬到 Settings，Profile 变成纯粹的"看我自己"的身份镜像。代价是有人可能在 Profile 找不到修改入口，用链接和引导来补偿。

### Terrain 组件抽共享
**决策**: 把等高线地形图从 Profile.jsx 的 inline 代码抽成 `src/components/Terrain.jsx`，Profile / Home / Onboarding 三个页面共用。
**原因**: 同样的几何代码在三个文件里各一份很快会发散。抽成组件后一处改动三处生效，三种 variant（default/mini/hero）覆盖不同使用场景。

### PaperRead 三栏 + PDF Viewer
**决策**: 论文详情页从「左文章右对话」双栏改成「左 TOC / 中 PDF / 右记忆通道」三栏，引入 pdfjs-dist 渲染 PDF。
**原因**: 原来的右侧对话栏让 papermind 看起来和任何 AI 阅读器一样。把聊天压到底部可唤起的 sticky bar，腾出右栏给记忆内容（为什么推这篇 / 你追问过的句子 / 画像更新预告），才能让用户感受到"这个 AI 在帮我记着我和这篇论文发生过什么"。
**关键取舍**: 选 pdfjs-dist 而不是 react-pdf，前者更可控，worker 配置用 Vite 的 `?url` 语法一行搞定。PDF CORS 问题通过 `/api/pdf-proxy` 解决，后端流式转发。

### cache["fetching"] 超时解锁
**决策**: 新增 `fetching_since` 时间戳，超时 5 分钟或 `force_fetch=true` 时强制重置 fetching 状态。
**原因**: 后台线程卡在网络请求时，`fetching` 标志永远为 True，前端轮询无法解除。重启服务器是当时的唯一解，加超时逻辑后前端可以自救。

## 2026-05-07

### 后端拆分：先拆业务 service，再拆 routers
**决策**: 先从 `api.py` 中渐进式抽出 `llm_router.py`、`search_service.py`、`memory_service.py`，暂不急着把 FastAPI 路由按文件全面拆开。
**原因**: 当前产品仍在验证体验，全面拆路由会带来较大改动面，但用户短期感知不明显。LLM、搜索推荐、记忆系统是变化最频繁、最容易互相影响的业务核心，先抽 service 能立刻降低维护风险，同时保持外部 API 不变。
**边界**: `api.py` 继续负责 HTTP 请求/响应、用户缓存、后台线程入口和收藏/导出等接口；业务 service 负责可独立测试的核心逻辑。
**护栏**: 新增 `tests/test_backend_guards.py`，先覆盖 LLM 无配置兜底、检索词锚点、排除词过滤、低价值论文过滤、推荐过滤链路和画像保存不覆盖后端记忆字段。后续继续拆 routes 前，先扩展这些测试。
