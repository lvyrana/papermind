# 产品决策记录

记录 PaperMind 开发过程中的重要产品决策，包括选择的方案、放弃的方案和原因。

---

## 2026-04-11

### 系统观察摘要退出搜索召回，只保留在手动修正后的理解层
**决策**: `interests_summary` 不再参与关键词生成；论文打分和 AI 解读只在 `interests_summary_is_manual = '1'` 时参考“用户修正后的偏好”。  
**原因**: 系统观察摘要来自行为归纳，容易很散，适合帮助模型理解用户，但不适合主导“搜什么”。如果把它接进检索召回，系统容易把“看过很多”误当成“现在要搜很多”，导致方向发散。  
**补充**: 用户手动编辑或手动清空摘要后，保存画像不会立刻触发自动重生成，避免用户修正被系统覆盖。

### 收藏页改为紧凑行列表，优先支持“回看我的笔记”
**决策**: 收藏页不再按时间大卡片分组展示，改为更紧凑的单列行列表，并增加“有笔记”筛选；删除按钮在移动端保持可见。  
**原因**: 当前阶段用户的核心任务不是“浏览漂亮的论文墙”，而是“快速找到自己写过笔记、想回看的内容”。大卡片和 hover-only 操作在手机上都不够顺手。  
**取舍**: 先优先回看效率和移动端可操作性，知识图谱/灵感库形态留待后续单独视图探索，而不是继续叠加在收藏列表上。

### 研究画像改为“长期偏好 + 系统观察”工作台
**决策**: 不再维持旧版逐项表单，改成新的研究画像工作台：研究方向、方法兴趣、随手补充、排除内容、学科领域，加上一块可查看和编辑的“系统观察摘要”。  
**原因**: 用户反馈旧画像页“像设置表单，很难写”，尤其研究经历和追踪主题不容易维护。新结构把长期偏好和系统自动学习结果放到同一页，填写成本更低，也更接近真实研究工作流。  
**代价**: 页面结构更复杂，字段语义需要后端同步梳理；如果不及时清理旧字段，容易出现前端不再暴露但后端仍在读取的历史状态。

### 收藏页回归 v0.5 信息结构
**决策**: 收藏页本轮不继续往“复杂灵感库”方向扩张，回到更接近 v0.5 的信息结构。  
**原因**: 用户明确反馈“内容更多了也更丑了，看笔记也不方便”。在当前阶段，比起继续加功能，更重要的是保住可读性和回看笔记的效率。  
**取舍**: 先稳定收藏体验，知识图谱/灵感库能力留待后续以独立视图或模式切换的方式尝试，而不是直接叠加在收藏列表上。

### 用户归属校验改为严格匹配，不再兼容空 user_id
**决策**: 收藏详情、删除收藏、保存笔记、AI 对话、对话总结等接口统一改为严格按 `user_id` 校验归属。  
**原因**: 兼容空 `user_id` 的旧策略虽然能照顾早期测试数据，但会让多人测试环境下的笔记、对话、导出都存在串数据风险。既然项目已经建立了设备级 UUID 隔离，后续实现应围绕严格归属校验收紧。  
**补充**: 为避免旧数据丢失，先做本地历史数据认领迁移，再切到严格校验。

### 搜索策略先回到 v0.5 主路径，再增量接入方法兴趣
**决策**: 放弃一度尝试的“方法学扩展宽搜”方案，回到 v0.5 的主题检索主路径，只把 `method_interests` 作为搜索词生成和排序的辅助输入逐步接入。  
**原因**: 纯方法学召回和过宽兜底会让结果快速发散，用户实际反馈是“只想在原有体感基础上稍微参考方法兴趣”，而不是完全重写召回逻辑。  
**代价**: 当前 `method_interests` 已参与搜索，但权重仍需继续调；时间范围在 Semantic Scholar 路径上仍然是宽松近似，不是严格月窗。

### 标题翻译与摘要翻译保持一致
**决策**: PaperRead 页的论文标题加"中文/原文"切换按钮，样式与摘要翻译按钮一致（文字小、低调）。  
**原因**: 用户反馈"进入论文页后标题没有翻译选项"；保持交互一致性，不引入新的 UI 模式。  
**细节**: 首次点击时懒加载翻译，之后切换仅切换显示状态，不重复请求。

### Legacy 论文 ownership 检查兼容策略
**决策**: ownership 检查改为 `if paper.get("user_id") and paper.get("user_id") != uid` 而非 `paper.get("user_id", "") != uid`。  
**原因**: 早期收藏的论文 `user_id` 字段为空字符串，旧检查将其视为"属于别人"，导致 GET/DELETE/notes 全部返回 not found。兼容写法：有 user_id 才做归属校验，没有则视为公共数据可访问。  
**代价**: 没有 user_id 的论文任何用户都能访问。可接受，因为这些都是早期测试数据。

### 对话失败反馈：显式错误而非静默失败
**决策**: "将对话保存为笔记"按钮的所有失败路径（自动收藏失败、API 错误、网络错误）均显示具体中文错误信息，不再 catch 后忽略。  
**原因**: 按钮点击后没有任何反应，用户不知道是成功还是失败；显式错误让用户知道该怎么做（如"请先手动收藏"）。

### Library 搜索与筛选：前端实现，不走后端
**决策**: Library 页的全文搜索、分类筛选、排序全部在前端完成，不新增后端接口。  
**原因**: 收藏数量有限（几十到几百篇），前端过滤够用；不值得为此加后端复杂度。  
**代价**: 超大收藏量时性能会下降，但对个人研究工具场景不是问题。

### Profile 两层结构：基础信息 vs 当前专注
**决策**: Profile 页拆为"基础信息"（相对稳定：研究方向、阶段、经验、学科）和"当前专注"（频繁变化：研究问题、目标、排除领域）两块。  
**原因**: 用户反馈画像维护成本高，一屏全部展示容易混淆稳定信息和动态信息。分层后用户知道"当前专注"要经常更新，"基础信息"不常动。

## 2026-03-31

### 无登录 + 设备隔离方案
**决策**: 不做用户登录，用 UUID 存 localStorage 实现设备隔离。  
**原因**: 测试阶段降低摩擦，用户打开即用。每台设备有独立数据，隐私更好。  
**代价**: 换设备或清除浏览器数据会丢失记录；无法跨设备同步。  
**未来**: 正式版考虑加可选登录（微信/邮箱），登录后可绑定设备 UUID 迁移数据。

### 内置 API Key，用户无需配置
**决策**: 把 API Key 放在后端 .env，前端 Settings 页简化为说明卡片。  
**原因**: 测试阶段朋友打开直接可用，不用被 API 配置劝退。  
**代价**: API 消耗由作者承担；不适合大规模开放。  
**未来**: 正式版开放自定义 API 配置，或转为订阅模式。

### LLM 优先级：阿里云通义 → GLM → DeepSeek
**决策**: 阿里云 DashScope 作为主力（免费额度大，模型质量好），后两者兜底。  
**原因**: 阿里云赠送了大量免费 token，qwen-flash/plus 质量明显优于 GLM-4-flash。  
**备注**: 可在 .env 中切换 QWEN_MODEL，目前测试 qwen-flash，生产考虑 qwen-plus。

### Rate Limit：普通用户 20次/天，owner 不限量
**决策**: 每人每天最多 20 次论文推荐，OWNER_UID 设备不限量。  
**原因**: 防止 API 被滥用；作者本人测试需要不限量。  
**实现**: .env 中设置 OWNER_UID，Settings 页展示设备 ID 方便复制。

## 2026-03-30

### 论文搜索：LLM 生成关键词，替代硬编码
**决策**: 不再硬编码 COPD 关键词，改为根据研究者画像 + 收藏历史，让 LLM 生成 PubMed 检索词。  
**原因**: 硬编码无法适应不同研究方向；画像驱动的检索结果更个性化，随使用积累越来越准。  
**效果**: 切换研究方向后，推荐论文立即跟随变化。

### 动态分类：LLM 打分 + 生成标签，替代硬编码分类
**决策**: 移除固定的六个 COPD 分类，改为 LLM 对每批论文打分（0-10）+ 动态生成分类标签。  
**原因**: 固定分类换了研究方向就失效；动态标签更贴合实际内容。  
**代价**: 每次推荐多一次 LLM 调用；标签命名不完全一致。

### 排除领域：双重过滤
**决策**: 排除领域在两处生效：① 生成关键词时不产生相关检索词；② 打分时强制 0 分。  
**原因**: 单一过滤不够——搜索词层面不拦截会抓来不想看的论文；打分层面不强制会仍然混入低分但非零分的内容。

### 异步抓取 + 前端轮询
**决策**: `/api/papers` 触发后台线程抓取，立即返回 `{loading: true}`，前端每 3 秒轮询。  
**原因**: 原来的同步模式下，用户离开首页会中断 fetch，回来后重新触发，浪费次数且体验差。  
**效果**: 抓取在后台完成，导航到其他页面再回来，结果已就绪。

## 2026-03-29

### 视觉风格：暖色系 + 呼吸感动画
**决策**: 主色调用暖白/米色/coral，避免大面积渐变，动画用 breathe-in 轻微入场。  
**原因**: 学术工具不应压迫感太重；暖色调比冷蓝色更有"陪伴感"。  
**原则**: 渐变只用于强调元素，不用于背景大面积铺色。

### 摘要翻译：按需触发，不自动翻译
**决策**: 摘要默认显示英文原文，点击"译"按钮后才请求翻译，可切换原文/译文。  
**原因**: 研究者通常能读英文摘要；自动翻译会消耗大量 API；按需触发节省资源，也不干扰习惯读原文的用户。

## 2026-03-27

### AI 解读的 relevance 字段
**决策**: relevance 只说"这篇论文的哪个发现/方法能给研究者带来什么具体启发"，不罗列研究者画像关键词。  
**原因**: 早期 GLM-4-flash 会把用户所有研究兴趣都列一遍，显得机械；用户明确反馈"笨笨的"。  
**Prompt 要点**: 明确告知"不要罗列研究者画像关键词"，且"如果论文没有直接涉及某个方向就不要提"。
