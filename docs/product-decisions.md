# 产品决策记录

记录 PaperMind 每一个关键设计决策的"为什么"，而不只是"做了什么"。

---

## 为什么叫 PaperMind

项目最初叫 copd-research-weekly，是给 COPD 领域做的周报工具。后来意识到关键词和领域都是配置化的，任何学科都能用。PaperMind 取 Paper（论文）+ Mind（记忆/思维），一词包含了文献追踪和对话记忆两个核心能力。

## 为什么首页叫"研究空间"而不是"论文列表"

普通学术工具一上来就是搜索框+表格，很工具感。但研究者打开一个每日助手，应该有被理解的感觉。所以首页用问候语（"早上好，研究者"）+ 上次在读 + 为你推荐，营造"你的研究伙伴已经帮你准备好了"的感觉。

## 为什么要做"研究者画像"

没有画像，推荐就是纯关键词匹配。有了研究方向、排除领域、当前目标，AI 能写出"这篇论文的 XX 方法可以迁移到你正在做的 YY"这种真正有用的解读。这不是功能堆砌，是让推荐从"相关"变成"有启发"。

## 为什么推荐结果要写"为什么与你相关"

学术工具给你 50 篇论文让你自己判断，但研究者时间有限。一句"这篇和你的研究方向的交集在于……"能帮你在 3 秒内决定要不要读。这是从"给你信息"到"帮你判断"的跨越。

## 为什么做对话式阅读而不是只给摘要

摘要是单向的。但读论文时真正的思考发生在追问中："这个方法靠谱吗？""样本量够吗？""和我做的有什么区别？"。对话式阅读把 AI 变成一个懂论文内容、也懂你研究背景的讨论伙伴。

## 为什么对话要有记忆

因为研究是连续的。你上周和 AI 讨论了一篇方法学论文，这周读到相关论文时，你不想从零开始解释自己的背景。对话记忆让每一次讨论都能积累，让 AI 越来越了解你。

## 为什么整体风格选温暖而不是科技炫酷

市面上的学术工具几乎全是冷色调+极简工具风。但科研是孤独的，尤其是独立研究者。PaperMind 用暖色系（cream/coral/mint）、圆角卡片、手感动画，想传达的是：这不只是一个工具，是一个陪你读论文的温柔空间。

## 为什么加呼吸感动画

论文列表如果一次性全部出现，信息量会让人焦虑。逐条呼吸入场（淡入+上移+微模糊）让加载过程像翻书一样，一篇一篇出来，节奏从容。这是从"数据展示"到"阅读体验"的细节。

## 为什么导出到 Zotero 而不是内置文献管理

PaperMind 的定位是"发现+理解+思考"，不是要替代 Zotero。研究者已有的文献管理流程不应该被打断。一键导出 RIS/BibTeX 让 PaperMind 成为现有工作流的入口，而不是孤岛。

## 为什么新用户引导要以"记忆"而不是"搜索"为核心卖点

测试反馈显示新用户打开首页完全不知道这是什么。我们有两种方向可以讲：一是"帮你搜论文"（功能层），二是"有记忆的助手"（价值层）。搜论文工具太多，但一个"告诉它一次、永久记住你、越用越懂你"的助手是有差异化的。引导页改为以"有记忆的学术文献助手"为标题，三个价值点也围绕记忆展开：记住你是谁 / 每天帮你过滤 / 沉淀你的思考。

## 为什么在画像未填时锁住"重新抓取"而不是只提示

第一批测试中有用户在画像空着的情况下连续点"换一批"，把一天的推荐配额（8批）全部消耗，却一篇论文没看到。根本原因：没有画像时检索词质量极差，大概率 0 结果或返回无关论文，但系统仍然扣配额。决定直接在 UI 层拦截：画像未填时隐藏"重新抓取"，空状态按钮变为"先填写研究方向"跳转 Profile。硬拦截比软提示更能防止这种情况。

## 为什么用专属链接而不是账号体系来支持跨设备访问

测试用户在电脑端测完后想在手机继续用，但现有设备 ID 机制要求手动复制粘贴一长串 UUID。登录体系需要邮箱/密码，成本高且目前用户量小。折中方案：生成带 uid 参数的专属链接（`?uid=xxx`），收藏书签或发给自己即可在任何设备无缝继续，无需理解技术细节。

## 为什么在后端而不只在前端拦截空画像的推荐请求

测试截图显示用户在画像为空的状态下触发了 8 次推荐抓取，每次 0 组查询、0 篇结果，但 8 次配额全部被扣光。前端引导页（画像空时显示 Onboarding）是第一道防线，但它只在"无缓存论文"时生效，存在绕过路径（如缓存被清、localStorage 问题）。后端守护才是最后的保障：在即将 `increment_rate_limit` 之前检查画像是否有任何实质内容，若完全为空则拒绝并返回 `needs_profile: true`，配额不动。两道防线互补：前端负责引导体验，后端负责配额安全。

## 为什么 LibraryDetail 用轮询而不是 WebSocket 等待 AI 解读

手动添加论文后，后台 AI 解读需要 5-15 秒完成。WebSocket 实现复杂、部署需改 nginx 配置。简单轮询（4s 间隔，有结果即停）足够解决这个低频场景，且实现成本极低。加"AI 解读生成中…"占位是关键——没有它，用户看到空白区域会以为功能出错。

## 为什么用模式匹配清除 localStorage 而不是维护一个固定列表

`paper-notes-*`、`paper-chat-*`、`paper-bookmark-*` 的 key 是动态拼接 pmid/paper_id 的，数量不定。维护一个固定列表意味着每次新增 localStorage key 都要同步更新清理列表，容易遗漏。改为遍历 localStorage 全量匹配前缀，一次清干净，且后续加新 key 不需要改清理逻辑。

## 为什么解读轮询超时选 60s 而不是更长

AI 解读单篇论文正常耗时 3-10s，即使后台排队也不应超过 30s。60s（15 次 × 4s）已留有充足余量。超过这个时间说明后台大概率已失败，继续轮询只会浪费请求、让用户看到永不停止的转圈。超时后静默停止比报错更合适——用户可以稍后刷新重试。
