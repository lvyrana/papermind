# Claude Handoff: Memory System v1

## 1. 先看这个，避免再次误解

这次改的不是“把旧 interests_summary 再优化一下”。

这次改的是：

**从单段系统观察摘要，过渡到双层记忆系统。**

核心结构：

- `memory_core`
- `memory_recent`

不要再把它理解成“旧摘要换个名字”。

## 2. 绝对不要回退的原则

### 2.1 观察记忆不参与检索

这是硬边界。

观察记忆：

- 不参与 query 生成
- 不参与搜索召回

观察记忆只用于：

- AI 对话
- 论文中文解读

当前也不要再把它加回分类。

### 2.2 学科领域不参与检索生成

`discipline` 仍然保留，但它已经退出检索生成。

不要再通过任何隐式兜底，把 `discipline` 偷偷塞回 query。

### 2.3 recent 不要因为画像变化就清空

当前产品讨论已经收敛到：

- 测试期用户会频繁改画像
- recent 不能一改画像就“失忆”

所以：

- 不要恢复“focus_areas 一变就清空 memory_recent”这种逻辑

### 2.4 core 不是只能手动，也不是只能自动

当前方向是：

- 首次自动生成 `memory_core`
- 后续保留系统低频自动修正
- 同时允许用户手动点击“更新到长期画像”

也就是说：

**自动和手动并行，不是二选一。**

## 3. 旧版到底是什么

旧版只有：

- `interests_summary`

它的特点是：

- 长期偏好和近期变化混在一段文本里
- 用户很难理解它到底是“稳定画像”还是“最近观察”

旧版如果用户手动编辑或删除摘要：

- 会把 `interests_summary_is_manual` 设成 `1`
- 后端自动生成会跳过，不覆盖

但是：

- 一旦画像变化
- 后端会清空摘要并重置手动状态

所以旧版并不是“永久手动锁”。

## 4. 这轮最终想实现的是什么

### 4.1 memory_core

长期稳定骨架。

特点：

- 首次保存画像后自动生成一版
- 供对话和解读使用
- 后面可低频自动修正
- 用户也可以手动确认吸收 recent

### 4.2 memory_recent

近期关注变化。

特点：

- 基于最近行为信号
- 表示“最近新增了什么”
- 不参与检索
- 不应因为改画像直接清空

### 4.3 merge-to-core

用户点击按钮后：

- 把 `memory_recent` 吸收到 `memory_core`

不是让用户自己去编辑一段长期画像文本。

## 5. 当前 UI 目标

画像页第一版只需要简单做：

- 一个“长期研究画像”只读块
- 一个“近期关注变化”只读块
- 一个按钮：`更新到长期画像`

不要再回到旧版那种：

- 一个大文本框
- 用户直接编辑“系统观察摘要”

## 6. 当前实现时最容易犯的错

### 错误 1：把 memory 再偷偷加回检索

这是最需要避免的。

如果 memory 影响检索，会再次把召回带偏。

### 错误 2：把 recent 当成“每次都重写的新摘要”

recent 的目标不是“重新写一段最近总结”，而是：

- 保留已有 still-valid recent
- 再吸收新增观察

允许压缩，但不要无故丢失近期仍有效的变化。

### 错误 3：把 core 完全锁死成手动

用户的真实意思不是：

- `core` 以后只能手动更新

而是：

- 系统可以自动生成和低频修正
- 但用户也要有手动确认入口

### 错误 4：把 recent 因画像变化清空

这在测试期会让系统显得非常不稳定。

### 错误 5：把分类继续依赖旧摘要

当前方向已经明确：

- 分类只看用户明确画像
- 不再看旧 `interests_summary`

## 7. 当前实现后的测试优先级

如果继续做，不要先急着大改结构。

先测：

1. 新用户首次保存画像后，`core` 是否自然
2. 用户继续阅读/收藏/提问后，`recent` 是否自然
3. 点击“更新到长期画像”后，`core` 是否像长期骨架
4. 对话和论文解读是否真的更贴合用户

## 8. 当前可以继续做，但还没完全定死的点

- `recent` 观察窗口到底是 7/14/21 天
- 事件阈值是否合适
- auto-refresh `core` 的节奏是否合适
- 后续何时删除旧 `interests_summary` 兼容层

## 9. 你如果继续改，请优先遵守这个顺序

1. 不破坏“记忆不参与检索”的边界
2. 不破坏 `core + recent` 的双层结构
3. 不恢复“画像变化就清空 recent”
4. 不把 UI 改回单一可编辑摘要框

## 10. 一句话总结

请把当前系统理解成：

**一个处于第一版验证期的双层记忆系统，而不是旧摘要机制的轻微修补版。**
